# Remote auth & localhost forwarding

The single most annoying thing about working on a remote Mac: a CLI or coding tool needs you to
log in through a browser, but it opens the browser **on the remote**, and the OAuth callback wants
to return to **`localhost`** — which, from your laptop, is the wrong machine. Same story for any
local dev server. This is what agent-mac-ops solves.

## Why it happens

`localhost` is per-machine. The dev server and the OAuth callback listener both bind the **remote's**
loopback. Copy the auth URL to your laptop and finish the login, and the provider redirects your
laptop browser to `http://localhost:PORT/callback` — your laptop's loopback, where nothing listens.

Two independent fixes, both built in:

1. **Port forwarding** makes your laptop's `localhost:PORT` *be* the remote's `localhost:PORT`.
2. **Browser handoff** makes URLs that a tool opens on the remote pop up on your laptop instead.

Together: run the auth command on the remote, the page opens on your laptop automatically, you
consent, the callback tunnels home, done — no Screen Sharing.

---

## Port forwarding

### Auto on connect
`FORWARD_PORTS` in `config.env` (e.g. `"3000 8080"`) are forwarded every time you connect with the
alias. Start a dev server on the remote at `:3000`, then open `http://localhost:3000` on your
laptop — it's the remote's server.

### Ad-hoc, mid-session (random callback ports)
Many OAuth flows pick a **random** callback port, so you can't pre-list it. Forward it on the fly
without reconnecting:

```bash
box-fwd 54321                 # forward a known port
box-fwd cancel 54321          # stop
box-fwd oauth '<paste auth URL>'   # parse the callback port out of the URL, forward it, AND open the URL here
```

`box-fwd` keeps its own short-lived SSH master connection, so it works regardless of your
`~/.ssh/config`. (Adding `ControlMaster auto` / `ControlPersist` for the host speeds up every
connection and is recommended, but not required.)

## Browser handoff (Tier 2 — open remote URLs on your Mac)

When enabled (`HANDOFF_ENABLED=true`), `./setup.sh remote` installs a small `open` shim at
`~/bin/open` on the remote, and the dev session puts `~/bin` first on `PATH` and sets `BROWSER` to
it. When a tool in that session opens an `http(s)` URL, the shim forwards it — over a **reverse SSH
tunnel** carried by your session (`-R`) — to a tiny listener on your Mac, which runs `open`.

Set the listener and its reverse tunnel up once on your Mac:

```bash
control/bin/install-open-listener.sh   # launchd agent: listener on 127.0.0.1:$HANDOFF_PORT
control/bin/install-handoff-tunnel.sh  # launchd agent: always-on, self-healing reverse tunnel
```

### How the pieces fit
```
remote dev session                              your Mac
──────────────────                              ────────
tool calls `open https://auth…?redirect_uri=…localhost:CBPORT…`
   └─ ~/bin/open shim ── curl localhost:PORT ──▶ (reverse -R tunnel) ──▶ open-listener.py
                                                                              │ 1. parse CBPORT from the URL,
                                                                              │    forward -L CBPORT to the remote
                                                                              └ 2. `open` → your browser
                                                                              ▲
auth provider 302 → http://localhost:CBPORT ◀── (forward -L CBPORT) ◀──────── your browser consents
```

### Auto-forward (no manual `box-fwd` for OAuth)
OAuth callback ports are usually **random**, so they can't be pre-listed in `FORWARD_PORTS`. The
listener handles this automatically: before opening a handed-off URL it scans for a `localhost:PORT`
(in `redirect_uri` or the callback path), and forwards that port to the remote first (via `box-fwd`,
fired async so the shim's curl still returns fast). By the time you finish consenting, the redirect
to `localhost:PORT` has a tunnel waiting. So a remote OAuth login is just: trigger it → consent on
your Mac → done. `box-fwd oauth <url>` remains the manual fallback for tools that bypass the shim.

### Security
- The listener binds **127.0.0.1 only** and accepts **http(s) only**.
- A shared `HANDOFF_TOKEN` (generated by `setup.sh`, baked into both the shim and the listener) is
  required, so a stray process on the remote's loopback can't make your Mac open arbitrary pages.
- The reverse tunnel is held open by a dedicated **self-healing launchd agent**
  (`install-handoff-tunnel.sh`), independent of any session: a re-dial loop reconnects within
  seconds of a network drop or remote reboot, and within ~a minute of a *silent* link death (laptop
  sleep/wake, Wi-Fi change) once its server-alive probe (`ServerAliveInterval=20 ×3`) times out. So
  the handoff keeps working even with no session connected. (The `-L` dev-server forwards still ride
  the on-demand session master and are only up while you're connected.)
- After a *silent* drop the re-dial can't re-bind the remote's port until the remote sshd releases
  the old forward. To bound that tightly, set `ClientAliveInterval 30` in the remote's
  `/etc/ssh/sshd_config`; otherwise it falls back to TCP keepalive, which can take minutes.

### Limitations
- The shim only catches tools that invoke `open` **by name** (PATH-resolved) or honor `$BROWSER` —
  which is most macOS CLIs and anything using Node's `open` or Python's `webbrowser`. A tool that
  calls `/usr/bin/open` by absolute path, or talks to LaunchServices directly, bypasses it; for
  those, fall back to copy-paste + `box-fwd oauth`.
- If the listener isn't running (or the tunnel is down), the shim opens the URL **on the remote** as
  a graceful fallback — nothing silently fails, you just land back in the Screen-Sharing case.

## Files

Move files to/from the remote without typing the host — `rsync -avzP` (resumable,
shows progress) with `REMOTE_HOST` filled from `config.env`:

```bash
<alias>-push ~/Downloads/report.json        # → remote home
<alias>-push ./dist ~/deploys/              # → a remote dir (last arg = dest)
<alias>-pull ~/Work/app/out.csv             # → cwd
<alias>-pull '~/logs/*.log' ./logs/         # quote globs so the REMOTE expands them
```

One path goes to a sensible default (remote home for push, cwd for pull); pass a
second to set the destination. Directory trailing-slash follows rsync's rule
(`dir/` copies contents, `dir` copies the dir). For ad-hoc one-offs, plain
`scp`/`rsync` against the host alias work too — these just save the typing.

## Clipboard images (screenshots → Ctrl+V on the remote)

macOS clipboards are per-machine, and a terminal forwards clipboard **text** only —
never image bytes. So a screenshot you copy on your laptop never reaches the remote,
and a tool running there (Claude Code, say) pastes from the **remote's** clipboard,
which never saw it. Three options, lowest-friction first:

- **Auto-mirror (zero friction):** `control/bin/install-clip-watch.sh` installs a
  small launchd watcher that polls your clipboard and, whenever a NEW **image**
  lands, pushes it to the remote's clipboard automatically. Copy a screenshot
  (`Cmd+Ctrl+Shift+4`), then `Ctrl+V` on the remote — nothing to run in between.
  Images only (text already pastes over the terminal); de-duped so a static image
  isn't re-sent; poll interval is `CLIP_WATCH_INTERVAL`. macOS has no clipboard-
  change *callback*, so polling is the only way. Stop it with
  `launchctl unload ~/Library/LaunchAgents/com.agent-mac-ops.clip-watch.plist`.
- **`<alias>-clip` (on demand):** the same push, run manually — good if you'd rather
  not have a background watcher. Copy the shot, run `<alias>-clip`, `Ctrl+V`.
- **Apple Universal Clipboard (native, no tooling):** if both Macs share an Apple ID
  with Handoff + Bluetooth + Wi-Fi on, copying on one puts it on the other's
  clipboard automatically, images included. Use this if it's reliable for you; the
  watcher is the deterministic, Apple-ID-free alternative.

All three need `pngpaste` locally (`brew install pngpaste`) except Universal
Clipboard. Text pastes fine regardless (the terminal forwards it).

## Editor — `code .` for the remote

You're used to `code .` opening your editor in the current folder. On the remote
that's a trap: it would open the editor **on the remote** (headless, or via Screen
Sharing). What you actually want is your **laptop's** Cursor/VS Code, in Remote-SSH
mode, pointed at the remote folder — editor UI local, files remote.

This rides the same handoff listener as the browser bridge (same tunnel, same token):

```bash
# inside the connected session (the `code .` equivalent — knows your cwd):
code-<alias>                 # open the current remote dir
code-<alias> apps/web        # a subdir
code-<alias> ~/other/repo    # an absolute/~ remote path

# from a LOCAL shell, without connecting a session first:
<alias>-code                 # opens your configured WORK_DIR on the remote
<alias>-code apps/web        # a path under WORK_DIR (or pass an absolute/~ path)
```

How it works: `code-<alias>` is a shim at `~/bin/code-<alias>` on the remote (pushed by
`./setup.sh remote`, next to the `open` shim). It resolves the target to an absolute
remote path and POSTs it to the listener on your Mac over the reverse tunnel; the
listener runs `cursor --remote ssh-remote+<host> <path>`. Cursor then opens its own SSH
connection to `<host>` and boots a `cursor-server` there — so the host must be a real
entry in your `~/.ssh/config` (it is, if that's how you connect with `<alias>`).

- **Editor choice:** `REMOTE_EDITOR` in `config.env` (`cursor` default, or `code` for
  VS Code). The listener also falls back to whichever of the two it can find.
- **Requirements:** the handoff listener (`install-open-listener.sh`) and the self-healing reverse
  tunnel (`install-handoff-tunnel.sh`) both running on your Mac — no active session needed — and
  Cursor/VS Code with the `cursor`/`code` CLI installed locally (Cursor: ⌘⇧P → "Install 'cursor'
  command").
- The remote shim only forwards a path — it can **only** ever open `ssh-remote+<your
  configured host>`, never an arbitrary destination, and it's token-gated like `open`.

### Make plain `code .` smart (optional)

`code-<alias>` is deliberately a distinct name so it never shadows the remote's own
`code`/`cursor` CLI. But if your remote is a headless always-on box you almost always
reach over SSH, you may want muscle-memory `code .` to *just work* — hand off to your
laptop when you're driving the box remotely, yet still open locally on the rare occasion
you're sitting at it.

Drop this into the **remote's** interactive shell rc (e.g. `~/.zshrc` on the remote). It
dispatches on the one signal that actually tracks "am I being driven remotely": whether a
connected session's reverse handoff tunnel is up. No tunnel → you're local → real editor.

```bash
# Smart code/cursor: hand off to the laptop when the handoff tunnel is up, else open
# locally. Guarded on the shim, so it's a no-op anywhere the shim isn't installed
# (e.g. your laptop — safe to keep in shared/synced dotfiles). Replace <alias>.
if [ -x "$HOME/bin/code-<alias>" ]; then
  _amo_handoff_up() { nc -z -w1 127.0.0.1 "${HANDOFF_PORT:-17999}" >/dev/null 2>&1; }
  _amo_is_wait()    { for _a in "$@"; do [ "$_a" = "-w" ] || [ "$_a" = "--wait" ] && return 0; done; return 1; }
  _amo_smart_edit() { local real="$1"; shift
    if ! _amo_is_wait "$@" && _amo_handoff_up; then "$HOME/bin/code-<alias>" "$@"; else command "$real" "$@"; fi; }
  code()   { _amo_smart_edit code "$@"; }
  cursor() { _amo_smart_edit cursor "$@"; }
fi
```

- **Why a shell function, not a `~/bin/code` wrapper:** functions load in interactive
  shells only, so non-interactive callers (git's `core.editor`, `EDITOR`, scripts) keep
  getting the real local binary. The `-w`/`--wait` carveout is a second guard for the same
  reason — a handoff returns immediately and can't block, so blocking invocations stay local.
- **Port check** is `${HANDOFF_PORT:-17999}` — set `HANDOFF_PORT` or edit the literal to
  match your `config.env`.

## Recipes

**Local dev server on the remote**
```bash
# in config.env: FORWARD_PORTS="3000"
box                       # connect (auto-forwards 3000)
# start your server on the remote at :3000, then on your Mac:
open http://localhost:3000
```

**OAuth login from a remote CLI — fully automatic (handoff on)**
```bash
box                                   # connect; listener already running on your Mac
<tool> login                          # on the remote; the consent page opens on YOUR Mac
# consent → callback returns to the remote listener via the forward → done
```
If the tool uses a random callback port and it isn't in `FORWARD_PORTS`, the consent page still
opens on your Mac; if the callback then fails, copy the URL and re-run via `box-fwd oauth` so the
port gets forwarded.

**OAuth login — manual (handoff off, or a tool that bypasses the shim)**
```bash
<tool> login --no-browser             # suppress the remote browser; it prints an auth URL
box-fwd oauth '<paste the URL>'       # forwards the callback port + opens the URL on your Mac
```

## Troubleshooting

| Symptom | Fix |
|---|---|
| Page opens on the **remote**, not your Mac | Listener not running (`control/bin/install-open-listener.sh`), or the tool bypasses the shim (absolute `/usr/bin/open`). Use `box-fwd oauth`. |
| `403 bad token` in the listener log | `HANDOFF_TOKEN` mismatch — re-run `./setup.sh remote` (re-pushes the shim) and `install-open-listener.sh` (re-reads the token). |
| `code-<alias>` says **"can't reach your Mac's listener (handoff tunnel down)"** | The reverse tunnel isn't up. It self-heals; if it's been down a while, on your Mac re-run `control/bin/install-handoff-tunnel.sh`. Inspect with `launchctl print gui/$(id -u)/com.agent-mac-ops.handoff-tunnel` and `~/Library/Logs/agent-mac-ops-handoff-tunnel.log`. (Cause is almost always the tunnel, not the listener — the listener is a separate always-on agent.) |
| Callback page hangs / "can't connect" after consent | Callback port not forwarded. Find the port in the auth URL and `box-fwd <port>` (or use `box-fwd oauth`). |
| `box-fwd` says `Control socket … No such file` | Network blip; it auto-starts a master. Retry once. |
